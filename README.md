# 코인 자동매매 백테스팅 시스템

변곡점(Swing High/Low) 기반 시장 구조 분석과 MTF(4h/1h) 추세 추종 + RSI 박스권
전략을 백테스트하고, 결과를 차트/테이블로 확인할 수 있는 시스템입니다. 매매 로직
전체 명세는 [`docs/spec.md`](docs/spec.md)를 참고하세요.

## 구조

```
backend/    FastAPI 백엔드 (백테스트 엔진, BingX 데이터 연동)
frontend/   React + Vite 프론트엔드 (백테스트 UI)
docs/       매매 로직 명세서
```

## 사전 준비물

- Python 3.12 이상
- Node.js 20 이상

## 백엔드 실행 (backend)

```bash
cd backend
python -m venv .venv

# Windows
.venv\Scripts\activate
# macOS/Linux
source .venv/bin/activate

pip install -r requirements.txt
```

환경 변수가 필요하면 `.env.example`을 복사해 `.env`로 만들고 값을 채워주세요
(시세 조회만 사용할 경우 BingX API 키는 없어도 동작합니다).

```bash
cp .env.example .env
```

개발 서버 실행 (`--host 0.0.0.0`으로 띄워야 같은 네트워크의 다른 기기에서도
접근할 수 있습니다. `127.0.0.1`로만 띄우면 로컬에서는 되지만 LAN의 다른
기기에서는 연결 자체가 거부되어 "네트워크 에러"가 나고 백엔드 콘솔에는
아무 로그도 남지 않습니다):

```bash
uvicorn app.main:app --reload --port 8000 --host 0.0.0.0
```

- 서버: http://127.0.0.1:8000 (로컬) / `http://<LAN IP>:8000` (같은 네트워크)
- 헬스체크: `GET /health`
- 백테스트 실행: `POST /backtest`

테스트 및 린트:

```bash
pytest
ruff check .
black . --check
```

## 프론트엔드 실행 (frontend)

```bash
cd frontend
npm install
npm run dev
```

- 개발 서버: http://localhost:5173 (백엔드가 http://127.0.0.1:8000 에서 실행 중이어야 합니다)
- `vite.config.ts`에서 `server.host: true`로 설정되어 있어 기본적으로 `0.0.0.0`에
  바인딩됩니다 (LAN의 다른 기기에서도 접근 가능).

빌드 / 린트:

```bash
npm run build
npm run lint
```

## 매매 로직 상세 (현재 구현 상태)

> 아래는 **지금 코드에 실제로 구현되어 있는 로직**을 정리한 것입니다.
> [`docs/spec.md`](docs/spec.md)는 최초 설계 명세서이고, 그 이후 백테스트로
> 검증하면서 몇 가지는 바뀌었습니다(예: 되돌림 진입 방식). 헷갈리지 않도록
> 이 섹션이 "지금 실제로 도는 코드" 기준의 최신 정리본입니다. 파라미터는
> 전부 `backend/app/services/backtest_engine.py` 상단 상수값입니다.
> 각 파라미터를 왜 이 값으로 정했는지는 [`docs/backtest_results.md`](docs/backtest_results.md)
> (백테스트 실험 기록)를 참고하세요.

### 1. 변곡점(Swing High/Low) 탐지 — `pivot.py`

- 캔들 i가 SH(스윙하이)가 되려면: 앞뒤 `lookback`개 캔들 전부보다 고가가 **엄격히** 높아야 함. SL(스윙로우)은 반대로 저가가 전부보다 엄격히 낮아야 함. 동률이면 인정 안 함.
- **순차 교차 규칙**: 확정된 변곡점은 SL→SH→SL→SH 순으로 교차해야 함. 같은 종류가 연속 후보로 나오면(예: SH 확정 후 SL 나오기 전에 더 높은 SH 후보 등장) 더 극단적인 값으로 갱신하고, 덜 극단적이면 무시.
- **아웃사이드 바(장대봉) 처리**: 한 캔들이 SH·SL을 동시에 만족하면, 양봉이면 SL→SH, 음봉이면 SH→SL 순서로 같은 캔들 안에서 순차 확정.
- lookback 값: **HTF(4h) = 2, LTF(1h) = 2** (`HTF_PIVOT_LOOKBACK`, `LTF_PIVOT_LOOKBACK`). 값을 키우면 스윙포인트가 덜 자주, 더 굵직하게 잡힘 (한때 HTF=4로 테스트했으나 백테스트 결과가 나빠져서 2로 원복).

### 2. 시장 구조 판단 — `market_structure.py`

가장 최근 확정된 4개 변곡점(SL 2개 + SH 2개, 시간순 교차)을 비교:

| SL 방향 | SH 방향 | 판정 |
|---|---|---|
| 상승(HL) | 상승(HH) | **상승추세 (Uptrend)** |
| 하락(LL) | 하락(LH) | **하락추세 (Downtrend)** |
| 그 외 (수렴/확장/동률 포함) | | **횡보 (Consolidation)** |

### 3. MTF 구조 — HTF 4h / LTF 1h / 진입정밀화용 1m

- **HTF(4h)**: 메인 추세 판단용. 백테스트 시작일보다 `HTF_LOOKBACK_BUFFER_DAYS`(180일) 앞서서부터 데이터를 가져와 추세가 처음부터 확정되어 있도록 함.
- **LTF(1h)**: HTF 추세 방향에 맞는 진입 시그널(OB, 아래 참고) 탐지용.
- **1m(1분봉)**: LTF에서 확정된 OB로 가격이 되돌아왔을 때, 그 1시간 구간 안에서 분 단위로 정밀 진입 시점을 잡는 용도로만 온디맨드로 조회.

### 4. 추세 진입 로직 (트렌드 트레이드)

1. HTF가 상승추세일 때 LTF에서 새 스윙로우(OB)가 확정되면(하락추세면 스윙하이), 그 즉시 진입하지 않고 **대기 상태(pending setup)**로 들어감.
2. OB 확정 이후 가격이 만들어내는 **가장 유리한 고점/저점(임펄스 극값)을 매 1시간봉마다 추적**.
3. 가격이 그 임펄스 극값에서 OB 쪽으로 **`RETRACEMENT_RATIO`(현재 0.5 = 절반 되돌림) 만큼만 되돌아오면** — OB까지 100% 되돌아올 필요 없음 — 해당 1시간 구간의 1분봉을 조회해서 **되돌림 방향으로 마감하는 첫 1분봉**을 찾아 그 종가/시각으로 진입.
   - 여러 비율(23.6%~100%)로 90일 백테스트 비교한 결과(RR 강제 전) 61.8%가 승률·손익·낙폭 모두 가장 좋아서 채택했었으나, 손익비 2:1 강제(아래 5번) 적용 후 BTC/ETH/SOL 1년 재검증에서는 50%가 세 심볼 모두 총손익·승률에서 61.8%를 앞서 50%로 변경함.
4. **무효화**: 진입 전에 1시간봉이 몸통 마감으로 SL을 넘어버리면 대기 상태 폐기. HTF 추세가 바뀌어도 폐기.
5. **SL**: OB 가격 기준 `SL_BUFFER_PCT`(0.1%) 버퍼만큼 바깥. **TP**: 진입가 확정 직후 그 시점 SL거리의 `TREND_TP_RISK_REWARD_RATIO`(현재 2배) 지점으로 강제 계산 (HTF 반대편 스윙이 아님).
   - BTC/ETH/SOL 1년 백테스트로 강제없음 vs 손익비 2:1 vs 3:1을 비교한 결과, 강제하는 쪽이 세 코인 모두 승률·낙폭이 개선됐고 2:1이 3:1보다 총손익도 더 좋아서 2:1을 채택함 (HTF 반대편 스윙은 너무 멀어서 도달 확률이 낮았음).
6. **청산/익절 체크**: 진입 이후에는 다시 1시간봉 기준으로, 캔들의 고가/저가가 SL 또는 TP를 건드리면 그 가격에 청산. 같은 봉에서 SL·TP가 동시에 걸리면 보수적으로 SL이 먼저 걸린 것으로 처리.

### 5. 횡보(박스권) 진입 로직 — RSI 기반, 변경 없음

HTF가 횡보(Consolidation)로 판정되면 LTF에서 RSI(14) 기반 박스권 매매 전환 (즉시 진입, 1분봉 정밀화 대상 아님):

- **롱**: 가격이 박스 하단(직전 HTF 스윙로우) 근처(0.5% 이내)이고, RSI가 30 이하에서 30 위로 재돌파할 때 진입.
- **숏**: 가격이 박스 상단(직전 HTF 스윙하이) 근처이고, RSI가 70 이상에서 70 아래로 재돌파할 때 진입.
- **SL**: 캔들이 박스 경계 밖에서 **몸통 마감**될 때(꼬리만 찌른 건 무시). **TP1**: 박스 중심선 도달 시 50% 청산 + 손절을 본절가로 이동. **TP2**: 박스 반대편 경계 도달 **또는** RSI가 반대편 극단(롱이면 과매수, 숏이면 과매도) 도달 시 잔량 전량 청산.

### 6. 포지션 사이징 & 리스크 관리 — `position_sizing.py`

- **고정비율 리스크 모델**: 매 거래마다 `risk_amount = 그 시점 자본 × risk_per_trade_pct(1%)`만 잃도록 진입수량을 역산 (`position_value = risk_amount / SL거리%`, `수량 = position_value / 진입가`). 손절폭이 좁을수록 포지션이 커짐 — 즉 승리 시 손익비가 매우 커질 수 있음 (실측: ETH 1년 백테스트에서 단일 거래가 계좌의 22%를 벌기도 함).
- **청산위험 필터**: `liquidation_buffer_pct`는 레버리지에서 자동 계산 (`(1/leverage) × 0.9`, 10배 기준 9%). 손절폭이 이 값을 넘으면 그 거래는 스킵. 레버리지를 바꾸면 이 필터 기준도 같이 바뀜.
- 기본 레버리지: **10배** (`leverage`), 리스크: **자본의 1%** (`risk_per_trade_pct`).

### 7. 데이터 소스 — `bingx_client.py`

- BingX 공개 API `GET /openApi/swap/v3/quote/klines` (서명 불필요). 응답은 최신순 내림차순 객체 배열이며, `limit`을 1440으로 요청해도 서버가 실제로는 1000개로 캡함 — 그래서 "요청한 개수보다 적게 왔으면 끝"이 아니라 **커서(가장 오래된 캔들의 시각)가 시작일에 도달할 때까지** 뒤에서 앞으로 계속 페이징하고, 마지막에 오름차순으로 정렬.
- 페이지 사이마다 1.1초씩 대기해 거래소 레이트리밋(1req/s)을 지킴.
- **1분봉은 최근 1년치만 보관됨** (실측: 365일 전은 데이터 있음, 730일 전은 없음). 1년보다 오래된 구간을 백테스트하면 추세 진입의 1분봉 정밀 진입 단계에서 데이터가 비어있어 진입이 안 잡히고, 영향받지 않는 박스권 전략만 거래를 냄.

### 8. 백테스트 실행 흐름 (비동기 Job)

- `POST /backtest/jobs` → 즉시 `job_id` 반환, 실제 계산은 백그라운드(`asyncio.create_task`)로 실행.
- `GET /backtest/jobs/{job_id}` → 진행 상태(`pending/running/completed/failed`), 진행률(%), 현재 단계 메시지를 폴링. 완료 시 전체 결과 포함.
- 진행률 단계: HTF 조회(0%) → LTF 조회(15%) → 시뮬레이션(30~100%, 캔들 순회하면서 갱신).
- `POST /backtest`(동기식, 완료까지 블로킹)도 스크립트/curl용으로 남아 있음.
- job은 프로세스 메모리에만 저장됨 (재시작하면 초기화, 1인 로컬 사용 규모라 충분).

### 9. 알려진 단순화 / 제약사항

- HTF/LTF 변곡점은 전체 구간을 한 번에 계산 후 "확정된 시점(index+lookback)"부터 안 것으로 취급. 실시간 봇처럼 미확정 변곡점을 매 틱 갱신하며 추적하지 않음 (명세서 6.2의 상태 지속성은 봇 단계에서 별도 구현 필요).
- 동시 포지션은 1개로 제한 (대기 중인 진입 셋업도 1개까지).
- 백테스트 구간 끝에 포지션이 열려있으면 마지막 캔들 종가로 강제 청산.
- 표본이 적은 기간(1~3개월, 수십 건)에서는 승률/손익이 소수의 큰 승리 거래에 크게 좌우되므로, 짧은 기간 비교 결과의 통계적 신뢰도는 낮음 — 되도록 6개월~1년 단위로 검증 권장.

## 사용 방법

1. 위 순서대로 백엔드와 프론트엔드를 각각 실행합니다.
2. 브라우저에서 http://localhost:5173 접속.
3. 코인 심볼(예: `BTC-USDT`)과 조회 기간을 입력하고 "Run Backtest"를 클릭합니다.
4. HTF(4h)/LTF(1h) 캔들 차트에 변곡점과 진입(EP)/손절(SL)/익절(TP)이 번호와 함께
   표시되고, 아래 거래 테이블과 요약 통계에서 결과를 확인할 수 있습니다.

## 같은 네트워크(LAN)의 다른 기기에서 접근하기

위 명령대로 백엔드/프론트엔드를 실행하면 둘 다 이미 `0.0.0.0`에 바인딩되어
있어 별도 설정이 필요 없습니다.

1. 백엔드를 실행한 PC의 LAN IP를 확인합니다 (`ipconfig`의 IPv4 주소, 예: `192.168.0.10`).
2. 다른 기기의 브라우저에서 `http://192.168.0.10:5173` 으로 접속합니다.
   - 프론트엔드는 접속한 주소(hostname)를 기준으로 백엔드 API 주소
     (`http://192.168.0.10:8000`)를 자동으로 사용하므로 별도 설정이 필요 없습니다.
   - 백엔드 CORS는 `localhost` / `127.0.0.1` / 사설 IP 대역의 `:5173` 출처만
     허용하도록 되어 있습니다 (`backend/app/core/config.py`의 `cors_origin_regex`).

> ⚠️ 이 설정은 **같은 LAN 안에서의 접근**을 위한 것입니다. 라우터 포트포워딩 등으로
> 인터넷에 직접 노출하는 것은 별개의 문제입니다 — 이 API는 별도 인증이 없으므로
> 실제로 외부 인터넷에 공개하려면 인증/방화벽 등 보안 조치를 먼저 추가하세요.
